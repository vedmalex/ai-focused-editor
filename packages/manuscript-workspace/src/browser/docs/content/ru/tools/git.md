Git хранит снимки книги во времени: закончили главу, сохранили срез — и к нему
всегда можно вернуться, даже если дальше текст сильно переписан. Две команды
этой страницы ничего не коммитят и не трогают историю сами — они только
готовят папку книги к тому, чтобы версии в принципе можно было вести. Дальше
за дело берётся полноценный инструмент контроля версий, встроенный в редактор
отдельно, — о нём ниже.

## Завести репозиторий для книги

Если папка книги ещё не под git, команда заводит его одним действием — без
терминала.

::action[Инициализировать репозиторий]{command="ai-focused-editor.git.initRepository"}

Команда лежит в самом низу меню **Manuscript** — верхнего меню редактора,
одной из последних (ниже неё — только пункт сброса раскладки окна для этой
папки).
Она берёт первую (и обычно единственную) папку открытого рабочего
пространства и запускает в ней `git init`. Если рабочая папка не открыта
вовсе, редактор честно попросит сначала открыть книгу как папку.

Что стоит знать заранее:

- **Проверка идёт по «внутри ли рабочего дерева», а не «есть ли `.git` именно
  здесь».** Если книга лежит внутри другой папки, которая уже сама под git
  (например, вы держите все книги в одном общем репозитории), команда увидит
  это как «уже репозиторий» и ничего не создаст — отдельного `.git` для книги
  не появится, даже если формально в самой папке книги его нет.
- **Без установленного git команда не работает** — ответ так и назовёт
  причину: «git init failed — is git installed and on PATH?». Никакой
  git-программы в комплекте редактора нет, нужен обычный системный `git`.
- Успешный ответ говорит, что индикатор git скоро появится в строке
  состояния внизу окна, — это уже не команда с этой страницы, а часть
  инструмента контроля версий, встроенного в редактор (см. ниже): он сам
  начинает следить за папкой, как только в ней появляется `.git`.

## Спрятать файл от версий

Тяжёлые или служебные файлы — экспортированный PDF, папка со сборкой,
чей-нибудь личный черновик — не всегда нужны в истории версий. Проще всего
исключить такой файл через контекстное меню дерева файлов: выделите файл или
папку правым щелчком и найдите пункт **Add to .gitignore** ближе к концу
списка.

Команда дописывает путь к файлу (относительно корня рабочего пространства) в
файл `.gitignore` — создаёт его, если такого файла ещё нет. Повторный вызов на
том же файле ничего не задублирует: редактор сначала проверяет, нет ли уже
такой строки (в том числе с ведущим слэшем), и просто сообщает, что запись
уже есть.

Честно об ограничениях:

- **Команда завязана на то, что выделено в дереве прямо сейчас** — вызвать её
  с этой страницы путеводителя нельзя, только из контекстного меню на самом
  файле.
- **Работает с одним файлом или папкой за раз** — выделить сразу несколько и
  добавить всех разом не получится.
- Если выбранный файл лежит вне открытой папки книги (случай с несколькими
  папками в одном рабочем пространстве), редактор откажется с сообщением, что
  файл вне рабочего пространства, — исключать чужие папки эта команда не
  умеет.

Кстати, ровно тот же приём — держать тяжёлые файлы вне истории — редактор уже
применяет сам в паре мест: например, папка со звуком нового набора расшифровки
уходит в `.gitignore` автоматически при создании набора (см.
:doc[Расшифровка аудио]{page="transcription"}), а диагностика книги отдельно
предупреждает, если рабочий ключ Groq не защищён от коммита, и предлагает
готовое исправление. Ручная команда с этой страницы нужна для всего
остального, до чего эти автоматические проверки не дотягиваются.

## Что дальше — сам контроль версий

Коммиты, ветки, история и сравнение версий в этом редакторе — не команды этой
страницы, а полноценный инструмент контроля версий, встроенный в редактор
отдельным пакетом (временный форк `@theia/git`, пересобранный под текущую
версию платформы — подробности в `packages/theia-git-fork/FORK.md`). Он живёт
в собственном пространстве команд `git.*`, отдельном от команд самой книги, и
эта страница его действия кнопками не дублирует — они привычны всем, кто
пользовался Source Control в VS Code, и вызываются так же.

Открывается панель **Source Control** — иконка на боковой панели или
сочетание `Ctrl+Shift+G` (на Mac `⌘⇧G`). В ней — список изменённых файлов со
стадированием (поставить/снять галочку изменения), поле для сообщения коммита
и кнопка **Commit**, сравнение версий файла (diff), переключение и создание
веток через быстрый выбор, синхронизация с сервером — Push, Pull, Fetch,
Stash. Всё это же продублировано в палитре команд под категорией **Git**
(`F1` → начните печатать «Git»). Коммит остаётся вашим осознанным шагом:
редактор его не делает вместо вас ни по одной из двух команд этой страницы.

Честно о зависимости: этой части редактора, как и командам выше, нужен
системный `git` на `PATH` — без него ни репозиторий не завести, ни панель
Source Control не заработает.

Если вы только осматриваетесь — начните с трёх шагов на странице
:doc[С чего начать]{page="start"}, а общий обзор возможностей лежит на
странице :doc[Путеводитель]{page="home"}.
