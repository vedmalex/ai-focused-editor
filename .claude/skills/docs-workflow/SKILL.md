# docs-workflow — документирование в путеводителе редактора

Тонкий оркестратор поверх встроенной системы документации редактора. Это **точка
входа + последовательность шагов**; вся механика (frontmatter, директивы,
`covers`, режимы `docs:*`, маршрут исключений, разбор падений) описана в
**каноне** — committed-гайде:

> **`docs/README.md`** (корень репозитория) — ЕДИНСТВЕННЫЙ источник истины.
> Читай его перед работой и обращайся к нему за каждой деталью. Этот skill
> НЕ дублирует его содержимое, только направляет.

Указатель на каталог контента: `packages/manuscript-workspace/src/browser/docs/README.md`.

## Interaction Model

- Type: non-interactive
- Why this type: процедурный оркестратор — фиксированная последовательность
  проходов (`docs:inventory` → писать → `docs:strict`); решения принимаются по
  канону `docs/README.md`, а не через опрос пользователя.
- State: stateless — состояние живёт в контенте и в гейте `docs:strict`, не в skill.
- Why state: skill только направляет; source of truth — файлы репозитория.
- Override: pass `INTERACTIVE=true` чтобы после нахождения непокрытых id / перед
  правкой очереди исключений сверяться с пользователем.
- In non-interactive mode: пройти сценарий до зелёного `docs:strict`, доложить итог.
- In interactive mode: остановиться на выборе носителя (шаг 2 сценария 1) и на
  оформлении записи в очередь исключений для подтверждения.

## Инварианты (не нарушать)

- Все команды `docs:*` запускаются **из пакета** `packages/manuscript-workspace`
  (`cd packages/manuscript-workspace` перед запуском).
- Контент — только `ru` (`content/ru/**/*.md`); en-заглушки не создавать (см. канон).
- Правь очередь `docs/coverage-exceptions.requests.jsonc`, **никогда** не
  `docs/coverage-exceptions.jsonc` (allowlist владельца) и не `*.generated.*`.
- Финиш ЛЮБОГО сценария: `docs:strict` зелёный (exit 0, uncovered=0, очередь=0)
  **и** визуальная проверка страницы в путеводителе.

## Сценарий 1 — задокументировать СУЩЕСТВУЮЩИЙ функционал

Цель: закрыть непокрытую команду/настройку продукта страницей.

1. **Снять инвентарь и найти пробел.** `cd packages/manuscript-workspace && bun run docs:inventory`, затем открой `docs/coverage-report.md` → секция `## Uncovered ids`. Это точный список того, что обязано быть описано (канон §1, §5).
2. **Выбрать носитель.** Существующая тематическая страница в `content/ru/**` или новая (создание страницы — сценарий 2). Новую генератор подхватывает автоматически, регистрировать не нужно.
3. **Описать возможность.** Опиши id прозой и заяви покрытие: точный `covers`, glob `{pattern, reason}` (потолок 8 id), либо директива `::action{command=}` / `::settings{query=<точный ключ>}`. Правила и порядок засчёта — канон §2, §3 (помни: префикс `query=` НЕ покрывает — только точный ключ).
4. **Если id не должен покрываться** (чужой namespace, рантайм-семейство, не-поверхность, отложено) — оформи запись в очередь `docs/coverage-exceptions.requests.jsonc` (kind: `external`/`dynamic`/`exempt`/`deferred`). Маршрут и виды — канон §4. Сам allowlist не трогай.
5. **Итерируйся мягко:** `bun run docs:dev` (сообщает, но exit 0), пока покрытие не завершено.
6. **Финиш:** `bun run docs:strict` зелёный + визуальная проверка. Падение — разбор по таблице симптомов в каноне §5.

## Сценарий 2 — разработать НОВУЮ документацию

Цель: новая страница-раздел путеводителя (например тематический хаб).

1. **Создать файл** `content/ru/<section>/<slug>.md`. Id страницы = путь без расширения (`content/ru/writing/formulas.md` → `writing/formulas`). Канон §2.
2. **Frontmatter** — только четыре ключа: `title` (обязателен), `order` (обязателен, целое ≥0), `section` (опц.), `covers` (опц.). Любой другой ключ валит сборку. Prose-only страница без команд/настроек легальна — `covers` опусти или `covers: []`. Канон §2.
3. **Содержание директивами** по нужде: `::action[..]{command=}`, `::settings[..]{query=}`, inline `:doc[..]{page="<id>"}` (атрибут `page=`, НЕ `link=`), контейнеры `:::steps` / `:::scenario` (последний только на `home`). Формулы KaTeX `$...$` — просто текст, инвентарю невидимы, покрывать нечем. Канон §2.
4. **Бэклинки/консолидация.** Если тема упоминается на других страницах — добавь на них указатель `:doc{page="<новый-id>"}`, чтобы разрозненные упоминания вели в единый хаб. `page=` обязан ссылаться на существующий id (иначе strict красный).
5. **Покрытие.** Если страница описывает команды/настройки — заяви их через `covers`/директивы (сценарий 1, шаг 3). Prose-only хаб покрытия не требует.
6. **Финиш:** `bun run docs:strict` зелёный + визуальная проверка (страница в навигации, примеры рендерятся, кросс-ссылки ведут куда надо). Падение — канон §5.

## Правило устойчивости

Гайд `docs/README.md` committed и лежит вне гитигноренного `memory-bank/` — он
переживает потерю memory-bank и свежий клон. Этот skill (`.claude/skills/`) тоже
committed. Гарантия «каждая возможность описана» держится гейтом `docs:strict`, а
не памятью агента. Порог входа низкий намеренно: создать `.md`, запустить три
команды; дисциплина существует, чтобы документация не устаревала молча.
