### overview
A "Proofreading" feature ported from ScanCheck into packages/manuscript-workspace, built entirely from existing repo conventions — NO new frameworks, NO backend HTTP image server, NO custom SplitPanel. Core shape: ONE ReactWidget (ProofreadingWidget: Navigatable + Saveable) opened by a priority-500 NavigatableWidgetOpenHandler on a book-native YAML sidecar file, exactly like entity-editor / excalidraw / office-preview. The widget renders three CSS-flex panes inside one shell: left = scan image via a FileService.readFile→base64 data URI (the repo's ONLY image convention, per semantic-markdown-preview-widget.ts), right = editable text (controlled <textarea> v1, mirroring entity-types-editor-widget.ts; optional imperative Monaco later), and — in translation mode only — a fixed read-only source pane between them (ScanCheck's #sourceTextViewer). A header row (like OfficePreviewWidget.renderHeader) carries prev/next navigation, a verified / needs-rework toggle, and an overall progress indicator. TWO modes (OCR proofreading vs translation proofreading) are a single Boolean off the sidecar's `mode` field — the SAME widget, differing only by which text file is editable, one extra read-only pane, mode-specific labels, and which AI actions are exposed (ScanCheck's isTranslationMode toggle, workspaceMode.js:1-3). AI runs through the existing ai-connect stack (generateWithFailover / getConfiguredProfile / getFailoverChain, per manuscript-workspace-contribution.ts:588-616) and results land as a diff+Apply through the existing ChangeProposalService.notifyReady — reuse, not reinvent. All pairing/progress/mode/sidecar logic is pure functions in src/common with bun tests; the widget/service are thin. Pairing is by basename across an images folder and a text folder declared in the sidecar, doctor-friendly under a new book-native proofreading/ area seeded by book-scaffold.ts.

### dataModel
Pure types + logic in a new src/common/proofreading-model.ts (mirrors ScanCheck src/shared/workspaceMode.js + utils.js, but book-native, workspace-relative paths). Types: `ProofreadingMode = 'ocr' | 'translation'`; `ProofreadingSet { mode; imagesFolder; textFolder; sourceTextFolder?; imageExtensions; textExtensions; pages: ProofreadingPage[] }`; `ProofreadingPage { base; verified; needsRework; aiVerdict? }` (aiVerdict optional, shaped like AliasCheckVerdict per manuscript-workspace-contribution.ts:1116 — a plausible per-page verdict record); `ProofreadingPair { base; imageUri; textUri; sourceTextUri?; missing }`. Pure functions: `isTranslationMode(set): boolean` (=== Boolean(set.sourceTextFolder), the direct analogue of workspaceMode.js:1-3 which keys off translationFolderPath); `getBaseName(name)` (lastIndexOf('.') split — carry ScanCheck's dotted-name behavior utils.js:11-13, tested explicitly for page.01.jpg→page.01); `matchPairs(imageNames, textNames, sourceTextNames, imageExts, textExts)` (one entry per image, basename Map lookup, localeCompare numeric sort, missing text ⇒ missing:true — port of utils.js matchFiles:59-92); `getEditableRelativePath(set, base)` (textFolder in OCR mode, textFolder is the translation file in translation mode — analogue of workspaceMode.js:27-34); `computeProgress(set): { verified; needsRework; total; percent }` (drives the header indicator and any tree/status chip). Defaults: imageExtensions ['.jpg','.jpeg','.png','.webp','.tiff','.bmp'], textExtensions ['.txt','.md'] — ScanCheck's defaults. Pairing runs in the BROWSER against FileService.resolve(folderUri) directory listings (same mechanism as manuscript-tree-model.ts scanKnowledge) — no node service needed.

### ocrMode
OCR proofreading mode (sidecar `mode: ocr`, no sourceTextFolder). Left pane: scan image from imagesFolder (data URI). Right pane: the OCR text file from textFolder is EDITABLE and Saveable — this is the artifact the writer corrects. Header shows verify (✅) and needs-rework toggles, prev/next, progress. AI actions row exposes exactly two, gated to this mode (ScanCheck #ocrAiActionsRow, UIManager.js:207-250): (1) Re-OCR — sends the scan image as a multimodal attachment plus a re-recognize prompt, returns a whole-page replacement, surfaced as ChangeProposal{originalText: current editor text, targetText: returned text} → notifyReady (ScanCheck reRecognizePage, TauriAiReviewService.js:705-754); (2) Proofread — runs per-paragraph structured proofreading (issues categorized orthography vs style with originalText/replacementText/explanation), then splices each accepted replacementText into the full text to build targetText and opens ONE ChangeProposal diff (ScanCheck proofreadParagraph/proofreadScope:647-962). v1 collapses ScanCheck's inline per-paragraph annotations into a single whole-page diff to reuse ChangeProposalService verbatim (see openQuestions on granularity). Missing-text pages (pair.missing) render with an empty editor and a "no OCR text" affordance; whether Re-OCR seeds a new text file is an openQuestion (OCR ingestion source).

### translationMode
Translation proofreading mode (sidecar `mode: translation`, sourceTextFolder set — the presence of sourceTextFolder IS the mode switch, mirroring ScanCheck's translationFolderPath keying, workspaceMode.js:1-3). THREE panes in the same widget: left = scan image (relabeled "Оригинал"/"Original"); a fixed read-only middle/under pane showing the ORIGINAL text from sourceTextFolder (ScanCheck's <pre id=sourceTextViewer>, index.html:199-202 — NOT editable, no Monaco/search, a plain scrollable pane); right = the TRANSLATION file from textFolder, EDITABLE and Saveable. Header verify toggle relabels the reject action to "Требует доработки"/"Needs rework" (ScanCheck delete→rework relabel, UIManager.js:207-250). AI actions row exposes the other two features (ScanCheck #translationAiActionsRow): (1) Translate — sends source text (and image attachment) + translate prompt, returns a whole-page translation → ChangeProposal diff into the translation file (ScanCheck translatePage:1008-1057); (2) Translation-QA — structured JSON comparing original vs translation (issues omissions/meaning/terminology + a correctedTranslation), applied as a whole-page ChangeProposal from correctedTranslation (ScanCheck verifyTranslation:1059-1119). GENUINE DIFFERENCES vs OCR mode, and the ONLY branches in shared code: (a) isTranslationMode true → render the extra read-only source pane and load sourceTextUri; (b) getEditableRelativePath returns the translation file, not the OCR file; (c) mode-specific nls labels; (d) which two AI action buttons show. Everything else (image loading, navigation, verified toggle, progress, Saveable, sidecar persistence) is shared. Note ScanCheck's documented gap: no line-level original↔translation merge/alignment; whole-page replacement only — v1 intentionally inherits this (translation-review-mode.md).

### sharedUi
One ProofreadingWidget (src/browser/proofreading-widget.ts): ReactWidget + Navigatable + Saveable, static FACTORY_ID, configure(uri) sets id/title/icon and kicks async load() of the sidecar + pair resolution. Registered by ProofreadingOpenHandler (src/browser/proofreading-open-handler.ts) extending NavigatableWidgetOpenHandler with canHandle returning a fixed 500 when uri basename === 'proofset.yaml' under proofreading/ (else 0) — identical to excalidraw-editor-open-handler.ts:15-27, so the two-pane view is default-open on the sidecar while raw YAML stays reachable via an explicit 'Open Raw YAML' command (editorManager.open bypassing priority, the entity-editor precedent). Wiring in src/browser/proofreading-frontend-module.ts: bind widget inTransientScope (one per URI), WidgetFactory toDynamicValue resolving + configure, OpenHandler inSingletonScope + bind(OpenHandler).toService, CommandContribution/MenuContribution; add {"frontend":"lib/browser/proofreading-frontend-module"} to package.json theiaExtensions (a new isolated entry beside excalidraw/office-preview, lines 57-90). Shared UI, mode-agnostic: header row (nav prev/next with ⌘←/⌘→ shortcuts, verify/needs-rework toggles with ⌘S/⌘⌫ per ScanCheck index.html:150-151,209-210, progress "12/40 verified · 30%"), left image pane (mtime-keyed base64 data-URI cache + 10MB single-image cap, copied from semantic-markdown-preview-widget.ts bytesToBase64), right editable text pane (controlled textarea v1). An injectable ProofreadingService (src/browser/proofreading-service.ts, following browser-manuscript-workspace-service.ts) is the SoT for pair resolution + progress + AI orchestration, so a tree section or status-bar chip can reflect verified counts (semantic-history-widget.ts STATUS_META is the per-file status-chip precedent). i18n: every label via nls.localize with ru + en keys (ScanCheck strings are Russian; provide both), matching change-proposal-service.ts's nls usage.

### aiWorkflow
Four AI features reused from ScanCheck, all routed through the EXISTING stack — no new AI client. In ProofreadingService, resolve profile+chain via this.aiProfilePreferences.getConfiguredProfile() / getFailoverChain() and call generateWithFailover(this.aiConnection, chain.length ? chain : [profile], { messages, parameters, logContext }, this.requestLog.createRecorder(commandId)) — the exact shape at manuscript-workspace-contribution.ts:588-616. Image-attachment features (Re-OCR, Translate, Translation-QA) attach the scan as a multimodal message part (base64, same bytes the left pane already loaded); verify the ai-connect message content shape supports image parts before Wave 7 (flagged in risks — ai-connect internals not inspected). Structured-output features (Proofread issues, Translation-QA) request JSON via parameters and parse into issue lists. RESULT APPLICATION is uniformly ChangeProposalService (change-proposal-service.ts): build ChangeProposal { uri: editable text file URI, originalText: current editor/model text, targetText: the AI whole-page result (Re-OCR/Translate replacement, or original text with accepted proofread/QA replacements spliced in), title }, then notifyReady(proposal, message) → Apply/Open-Diff; apply() re-checks originalText against the live model (drift guard) before writing. This gives the writer a Monaco diff + explicit Apply for EVERY AI action, reusing the exact flow the codebase already ships. Prompts: seed default prompt templates (ScanCheck appConfig.js:8-18 defaults) as reusable constants in common; optionally surface them as ai/prompts entries (custom-modes.yaml precedent) with per-feature profile override (ScanCheck resolveAiFeatureConfig, appConfig.js:670-700) — an openQuestion on storage location. Per-feature enable flags default on. Each AI call gets its own command id + createRecorder for request logging (AiRequestLogService).

### persistence
Verified-state + pairing config live in a comment-preserving YAML sidecar, one per proofreading "set", read/written via the `yaml` package Document API (parseDocument / document.set / document.delete) EXACTLY like entity-editor-widget.ts — so hand-written comments and unknown keys survive round-trips and can be surfaced read-only. New src/common/proofreading-sidecar.ts owns (de)serialize (paths stored workspace-relative like projectData.js:47-57 serializeProjectData) + an AJV schema class modeled on src/common/yaml-schema-validator.ts (YamlSchemaValidator, moved to common per git status) validating mode/folders/extensions/pages. Sidecar file: `proofreading/<set-slug>/proofset.yaml` (fixed basename the open handler recognizes). Content: mode, imagesFolder, textFolder, sourceTextFolder?, imageExtensions, textExtensions, and pages[] each with base/verified/needsRework/aiVerdict?. This is PER-SET (per-project) state — deliberately NOT ScanCheck's global app_config_dir/file-states.json (lib.rs:535-559), whose global-vs-per-project precedence ScanCheck itself left untraced; one store, in the sidecar, keyed by page base name (ScanCheck fileStates uses names not indices, WorkflowManager.js:527-581). The EDITED TEXT itself persists to the real text file (textFolder/<base>.<ext>) through the widget's Saveable.save() → FileService.write, and through ChangeProposalService.apply() (which uses MonacoTextModelService, so it needs a real file URI — satisfied). The widget marks itself dirty on text edits AND on verified-toggle changes; save() writes both the text file and the sidecar. ## RISKS
- ChangeProposalService.apply() writes the WHOLE document (getFullModelRange → targetText). ScanCheck's proofreading/translation-QA UX is inline per-paragraph/per-fragment annotations (TauriAiReviewService.js:647-962); collapsing to a single whole-page diff loses per-issue accept/reject granularity. v1 accepts this; a richer inline-annotation layer is future work (see openQuestions).
- generateWithFailover's message content shape for IMAGE attachments was not verified — ai-connect-theia internals were not inspected (research caveat). Re-OCR/Translate/Translation-QA all depend on multimodal parts; Wave 7 must confirm the accepted content-part shape (or a generateImage-adjacent path) before wiring, else those three features stall.
- No in-repo precedent embeds Monaco inside a ReactWidget; the textarea v1 avoids that risk but forfeits find/replace/minimap on the text pane. If Monaco is later required it is an imperative MonacoEditorProvider.createSimpleInline attach into a ref div with disposal lifecycle — must be prototyped, not assumed.
- Left-pane images are base64 data URIs with a 10MB single-file cap (semantic-markdown-preview-widget.ts). High-resolution book scans (multi-MB TIFF/PNG) may hit the cap or bloat memory across a large set; needs a cap/warn + possibly downscale strategy, and pairing over huge folders via FileService.resolve should be measured.
- If the editable text pane targets REAL content/chapter-NN.md (vs a working copy under proofreading/), edits interact with manifest.yaml reconstruction (manifest-reconstruction.ts) and git; a decision is needed to avoid surprising the manuscript tree / doctor. Recommend editing a working copy under proofreading/ in v1 (matches ScanCheck's separate text folder).
- getBaseName uses lastIndexOf('.') (ScanCheck utils.js:11-13): dotted names like page.01.jpg pair as base 'page.01'. Correct only if image and text files share the SAME dotted convention; mismatched conventions silently mis-pair — must be documented and tested (Wave 1).
- ScanCheck never resolved global (app_config file-states.json) vs per-project (.ocr_check fileStates) precedence. We deliberately choose per-set sidecar only; ensure no expectation of cross-book shared verified-state leaks in from the port.
- Widget marks dirty on BOTH text edits and verified-toggle; without the excalidraw version-baseline guard the initial paint could mark clean docs dirty. Reuse that guard (Wave 5) or risk spurious save prompts on every open.

## OPEN QUESTIONS
- FOLDER CONVENTION: confirm the book-native layout. Proposed: images under sources/scans/<set-slug>/ (fits existing sources/ scaffold) and a dedicated proofreading/<set-slug>/ area holding proofset.yaml + the editable text/ (+ source/ in translation mode). Alternative: reuse content/ for text. Which becomes the doctor-seeded standard?
- TEXT PANE TARGET: does the right pane edit the REAL manuscript text (e.g. content/chapter-NN.md) or a working copy under proofreading/? A working copy is safer (no manifest/git interference; matches ScanCheck's separate text folder) but requires an explicit promote-to-content step later. Owner's call.
- OCR INGESTION SOURCE: where do OCR text files come from initially — pre-generated externally and dropped into textFolder, or does Re-OCR create them on demand for pages with missing text (pair.missing)? This decides missing-text UX (empty editable pane + 'Run OCR' vs 'to-delete' auto-inclusion, ScanCheck utils.js:101-127).
- TRANSLATION SOURCE ORIGIN: in translation mode, is the read-only original text a separate sourceTextFolder, or is it the OCR output of the SAME book's OCR set (chaining OCR→translation)? And should the scan image still show in translation mode (ScanCheck: yes, relabeled 'Оригинал')?
- AI ISSUE-APPLICATION GRANULARITY: accept v1's whole-page ChangeProposal diff (simple, reuses ChangeProposalService), or invest in ScanCheck-style inline per-paragraph/per-fragment accept-reject annotations (orthography/style categorized)? The latter is a materially bigger UI and not covered by the current diff+Apply reuse.
- PROMPT + PROFILE STORAGE: store the four feature prompts (reOcr/proofreading/translation/translationQa) as constants in common, or as first-class entries in ai/prompts/custom-modes.yaml (editable per book), and do we want ScanCheck's per-feature profile override (resolveAiFeatureConfig, appConfig.js:670-700) or just the single configured profile?
- VERIFIED-STATE SURFACING: should proofreading sets appear as a section in the manuscript tree (like Sources/Knowledge) with per-page verified chips (semantic-history STATUS_META precedent), and/or a status-bar overall-progress indicator? Affects whether ProofreadingService must be a shared SoT beyond the widget.
- GIT-DIFF PARITY: ScanCheck offers a git HEAD↔working diff of the text file (TauriGitService). We have theia-git-fork with DiffUris.encode + EditorManager.open (git-diff-contribution.ts). Include a 'Diff against git' action for the text file, or defer? Not required for the two proofreading modes.
- TEXT PANE EDITOR: ship the plain controlled <textarea> (v1, matches entity-types-editor-widget.ts) or invest early in an imperative Monaco embed for find/replace/highlighting? Recommend textarea first, Monaco only if a hard requirement emerges.
