# Путеводитель редактора — как добавить и поддерживать страницу

Это committed-гайд по системе документации редактора («путеводитель» — встроенная
в приложение справка). Он лежит в `docs/` рядом с governance-файлами покрытия
(`coverage-exceptions*.jsonc`, `coverage-report.md`) и **намеренно вынесен из
`memory-bank/`**: `memory-bank/` гитигнорен, а этот гайд обязан пережить его
потерю и свежий клон. Механика документации иначе живёт только в комментариях
внутри `.mjs`-скриптов — этот файл собирает её в одном месте.

> **Язык контента — только `ru`.** Английский набор неполон и это штатно:
> резолвер падает на `ru` как на язык по умолчанию (`DEFAULT_DOCS_LANG = 'ru'`).
> Новые страницы пишите на `ru`; en-заглушки создавать не нужно.

Все команды `docs:*` запускаются **из пакета** `packages/manuscript-workspace`
(скрипты объявлены в его `package.json` и вызывают `bun scripts/...`).

```bash
cd packages/manuscript-workspace
```

---

## 1. Трёхпроходная схема

Документация — не просто набор `.md`. Сборка **инвентаризует** все команды и
настройки продукта и **проваливается**, если какая-то возможность не описана ни
одной страницей (или описана, но ссылается на несуществующую команду). Поэтому
работа идёт в три прохода.

### Проход 1 — снять инвентарь

```bash
bun run docs:inventory
```

`extract-feature-inventory.mjs` обходит исходники трёх пакетов
(`manuscript-workspace`, `ai-connect-theia`, `document-preview-theia`), собирает
все `command id` (в namespace `ai-focused-editor.` и `ai-connect.`) и все
`preference key` (`PreferenceSchema`) и пишет
`packages/manuscript-workspace/docs-inventory.generated.json` (гитигнорен). Это и
есть «список всего, что обязано быть описано». Непарсуемый ключ — не пропуск, а
**падение сборки** (exit 2): дыра в инвентаре — это молча снятая гарантия.

### Проход 2 — писать и править страницы

Страницы живут в
`packages/manuscript-workspace/src/browser/docs/content/<lang>/**/*.md`
(сегодня — только `ru`). Новую страницу генератор подхватывает **автоматически**,
регистрировать её где-либо не нужно — достаточно создать файл. Пока пишете,
итерируйтесь в мягком режиме (см. проход 3, `docs:dev`), чтобы не спотыкаться о
незавершённое покрытие.

### Проход 3 — проверить строго

```bash
bun run docs:strict     # = docs:inventory + docs:gen:strict; ГЕЙТ
```

`generate-docs-content.mjs --coverage=strict` читает инвентарь и все страницы,
генерирует `src/browser/docs/docs-content.generated.ts` (то, что показывает
приложение) и committed-отчёт `docs/coverage-report.md`, а затем **валит сборку**,
если:

- какая-то возможность продукта не покрыта ни страницей, ни исключением;
- кнопка ссылается на несуществующую команду, настройка — на несуществующий
  ключ, ссылка `:doc` — на несуществующую страницу;
- инвентарь устарел (fingerprint исходников разошёлся);
- очередь запросов исключений (`coverage-exceptions.requests.jsonc`) не пуста.

Для итерации без гейта:

```bash
bun run docs:dev        # = docs:inventory + docs:gen:warn; сообщает, но exit 0
bun run docs:gen:warn   # только генерация в мягком режиме (инвентарь не пересобирает)
```

**Важно про режимы.** «Мягкость» `warn` касается **только полноты покрытия**
(незакрытые id + непустая очередь). Битая ссылка (мёртвая кнопка, несуществующая
страница) — это дефект, а не незавершённость: он валит сборку в **обоих** режимах.
Отсутствие флага `--coverage` означает `strict` (fail-safe по умолчанию).

## 2. Анатомия страницы

### Путь = идентификатор

Id страницы — это её путь внутри `content/<lang>/` без расширения:

| Файл                                   | Id страницы     |
|----------------------------------------|-----------------|
| `content/ru/home.md`                   | `home`          |
| `content/ru/book/export.md`            | `book/export`   |
| `content/ru/writing/semantic.md`       | `writing/semantic` |

Именно этот id подставляется в `:doc[...]{page="book/export"}`.

### Frontmatter

Каждая страница начинается с YAML-frontmatter между `---`. Допустимы **только**
четыре ключа (любой другой валит сборку):

```yaml
---
title: Собрать книгу      # обязателен; непустая строка; заголовок в навигации и на странице
order: 30                 # обязателен; целое >= 0; позиция внутри секции (меньше — выше)
section: Книга            # опционально; группа в навигации (страницы без секции — верхнего уровня)
covers:                   # опционально; список покрываемых id (см. §3)
  - ai-focused-editor.bookBuild.buildMarkdown
---
```

Prose-only страница без единой команды/настройки — легальна: `covers` можно
опустить или оставить `covers: []` (пример — `content/ru/settings.md`).

### Директивы

Синтаксис — markdown-директивы (число двоеточий = уровень):

- **Кнопка команды** (leaf, два двоеточия):
  ```
  ::action[Собрать в Markdown]{command="ai-focused-editor.bookBuild.buildMarkdown"}
  ```
  `command=` обязан быть реальным id из инвентаря (иначе — мёртвая кнопка, fail).

- **Кнопка настроек** (leaf, два двоеточия):
  ```
  ::settings[Показать настройки подключения]{query="aiConnect"}
  ```
  `query=` — либо точный `preference key`, либо префикс на границе сегмента
  (открывает панель настроек с этой строкой поиска).

- **Ссылка на другую страницу** (inline, одно двоеточие) — атрибут `page=`, **не**
  `link=`:
  ```
  :doc[Собрать книгу]{page="book/export"}
  ```
  `page=` обязан быть id страницы, существующей в `content/ru`.

- **Контейнерные директивы** (три двоеточия) — блок между `:::name{...}` и `:::`:
  `:::steps{id="..."}` (чек-лист) и `:::scenario{page="..." icon="..."}` (крупная
  карточка-сценарий; разрешена **только** на `home`).

Формулы KaTeX (`$...$`, `$$...$$`) — это **не** директива и **не** команда: они
пишутся прямо в тексте и структурно невидимы инвентарю (у них нет command id /
preference key), поэтому и покрывать их нечем — страница о формулах описывается
чистой прозой.

## 3. Дисциплина `covers`

`covers` — это явное заявление страницы «я описываю вот эти возможности». Есть две
формы:

1. **Точный id** (строка):
   ```yaml
   covers:
     - ai-focused-editor.bookBuild.epub
   ```
   Покрывает ровно один id. Id обязан быть в инвентаре.

2. **Glob** (объект `{pattern, reason}`, `*` только в конце):
   ```yaml
   covers:
     - { pattern: "ai-focused-editor.excalidraw.*", reason: "все команды панели Excalidraw описаны в разделе «Диаграммы»" }
   ```
   `reason` обязателен и непуст. Glob — «потолок» (ceiling): один шаблон не может
   поглотить больше **8** id (N по умолчанию, `--glob-ceiling`), иначе сборка
   требует разбить его на точные id или отдельные страницы. Голая строка-glob
   (`"...*"` без объекта) запрещена — поглощение всегда несёт письменное
   обоснование.

### `query=` покрывает только точный ключ (ISS-096)

Кнопка `::settings{query="aiConnect"}` с **префиксом** ничего не покрывает — это
лишь affordance (открывалка панели), а не заявление о документировании. Покрытие
засчитывается только когда `query=` называет **точный** `preference key`
(`query="aiConnect.requestLog"`). Иначе ключи попадут в `Uncovered ids`, а сборка
напечатает advisory-NOTE с точной позицией. Чтобы описать группу ключей — назовите
их поимённо в `covers`, а кнопку оставьте с префиксом.

То же для `::action{command=}`: одна директива покрывает ровно тот id, который
называет.

Порядок засчёта в отчёте: точный `covers` → glob → директива-occurrence →
allowlist. Но **все проверки** (ceiling, детекторы протухания) работают на сырых
множествах совпадений.

## 4. Маршрут запроса исключения

Иногда id **не должен** покрываться страницей: чужая команда Theia, семейство
команд, регистрируемых в рантайме, служебный id, не являющийся пользовательской
поверхностью, или страница, которую только предстоит написать. Для этого есть
allowlist — но **правит его только владелец** (WP-2 / генератор). Контентная
работа исключения не вписывает напрямую.

**Правило:** добавляйте запись в **`docs/coverage-exceptions.requests.jsonc`**
(очередь запросов), а **не** в `docs/coverage-exceptions.jsonc` (сам allowlist).
Адъюдикация — это перенос записи из очереди в allowlist (или в страницу, или в
удалённую директиву); так `reason` автора никто не переписывает.

Запись очереди — это запись исключения плюс `requestedBy`:

```jsonc
{
  "requests": [
    {
      "pattern": "git.*",
      "kind": "external",
      "reason": "команды форка theia-git-fork — чужой namespace, вне обхода инвентаря",
      "requestedBy": "TASK-XXX",
      "added": "2026-07-23"
    }
  ]
}
```

### Виды исключений (`kind`)

Замкнутый набор из четырёх (проверяется в **обоих** файлах):

- **`external`** — чужой id (команда/настройка не нашего продукта). Может нести
  `usedBy: "content" | "code"` (`code` — id вызывается из нашего кода, но не
  является значением директивы). `external` **запрещено** наводить на наш
  собственный prefix.
- **`dynamic`** — семейство id, регистрируемых в рантайме по контенту
  пользователя (описывается прозой, не покрывается поимённо).
- **`exempt`** — id **не является** пользовательской поверхностью: нажать/вызвать
  нечего, описывать нечего. `reason` обязан это обосновать; «ещё не описано» —
  именно та причина, которую дисциплина отвергает.
- **`deferred`** — противоположность `exempt`: поверхность **реальна** и **будет**
  описана. Обязателен `deferredTo: "TASK-NNN"` (формат строгий) — задача, которая
  должна написать страницу. `deferred`-id **не** считаются описанными: у них своя
  строка в отчёте и своя секция `## Deferred coverage`.

### Как очередь взаимодействует с гейтом

- `strict` **отказывается собираться**, пока очередь непуста (§4.3) — сначала
  адъюдицируйте запросы.
- В `warn` ожидающий запрос вида `external`/`dynamic` временно удовлетворяет
  fence `command=`/`query=`, чтобы контент-автора не блокировала адъюдикация. В
  релиз это не пройдёт — `strict` всё равно не соберётся с непустой очередью.

## 5. Что делать, когда `docs:strict` падает

Сообщения печатаются в stderr. По типу:

| Симптом (сообщение) | Что произошло | Как чинить |
|---|---|---|
| `N uncovered id(s) — document them, exempt them, defer them to a task, or mark them dynamic` (exit 1) | Возможность продукта не описана ни страницей, ни исключением | Откройте `docs/coverage-report.md` → секция `## Uncovered ids`. Опишите id на подходящей странице (`covers` или директива), либо оформите исключение через §4 |
| `unknown command "X" in :action …` / `unknown settings query "X" …` / `directive :doc{page="Y"} … targets a page missing` (exit 1, **оба режима**) | Висячая ссылка: кнопка/ссылка ведёт в никуда | Исправьте id/`page=`, либо удалите директиву. Проверьте опечатку; для чужой команды — исключение `external` |
| `inventory is stale (source fingerprint mismatch) — re-run docs:inventory` (exit 2) | Инвентарь устарел (исходники изменились после последнего `docs:inventory`) | Пересоберите: `bun run docs:inventory` (или сразу `docs:strict` — он делает это первым проходом) |
| `N unapplied coverage-exception request(s) — adjudicate them into docs/coverage-exceptions.jsonc` (exit 1) | Очередь `requests.jsonc` непуста | Владелец адъюдицирует каждый запрос: перенести в allowlist, заменить страницей или удалить вместе с директивой. Очередь должна стать пустой |
| `stale exception "P" (kind=K) matches nothing in … — remove it` (exit 1) | Запись allowlist больше ни на что не указывает (id удалён/переименован) | Удалите протухшую запись из `coverage-exceptions.jsonc` |
| `inventory missing — run "bun run docs:inventory" first` (exit 2) | Не сняли инвентарь перед генерацией | Запустите `docs:strict`/`docs:dev` целиком (они снимают инвентарь первым проходом) |

Коды выхода: **1** — нарушение правила (контент/покрытие), **2** —
конфигурация/окружение (инвентарь, JSONC, root).

## 6. Ссылки на реальные файлы

- **`packages/manuscript-workspace/scripts/extract-feature-inventory.mjs`** —
  извлекатель инвентаря (проход 1). Namespace-фильтр, разбор команд/настроек,
  видимость непарсуемого.
- **`packages/manuscript-workspace/scripts/generate-docs-content.mjs`** —
  генератор + гейт покрытия (проходы 2–3). Режимы `strict`/`warn`, fences,
  `covers`, отчёт.
- **`packages/manuscript-workspace/src/node/docs/source-scan.ts`** — **единственный**
  обход исходников (roots, excludes, fingerprint); оба скрипта импортируют его,
  чтобы не разойтись.
- **`packages/manuscript-workspace/src/common/docs/docs-contract.ts`** — контракт
  данных (`DocsPage`, `DocsCoverageClaim`, `DEFAULT_DOCS_LANG`, `DocsContentProvider`).
- **`docs/coverage-exceptions.jsonc`** — allowlist (правит **только** владелец).
- **`docs/coverage-exceptions.requests.jsonc`** — очередь запросов (сюда пишет
  контент-автор, см. §4).
- **`docs/coverage-report.md`** — committed-отчёт покрытия (генерируется; читать,
  не править — детерминирован и недатирован, чтобы diff означал «покрытие
  изменилось»).

Страницы контента редактируются в
`packages/manuscript-workspace/src/browser/docs/content/ru/**` (см. указатель
`packages/manuscript-workspace/src/browser/docs/README.md`).

## 7. Правило устойчивости (durability)

- Этот гайд **committed** и лежит вне `memory-bank/` — он переживает потерю
  memory-bank и работает в свежем клоне.
- Инвентарь и отчёт — часть сборки: гарантия «каждая возможность описана»
  держится гейтом `docs:strict`, а не памятью автора.
- Контент — только `ru`; `en` неполон штатно (fallback на `ru`).
- Порог входа низкий намеренно: создать `.md`, запустить три команды. Дисциплина
  (`covers`, маршрут исключений) существует, чтобы документация не устаревала
  молча.
