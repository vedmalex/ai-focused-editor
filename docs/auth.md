# Защита браузерной версии общим секретом (опционально)

AI Focused Editor умеет закрывать браузерную серверную часть простой аутентификацией по
общему секрету. Механизм **по умолчанию выключен** и включается только когда секрет
действительно задан. Он не заменяет полноценную систему пользователей — это лёгкий барьер,
чтобы редактор, поднятый на LAN или за реверс-прокси, не был открыт всем подряд.

## Что важно знать сразу

- **По умолчанию отключено.** Без секрета и без флага `--auth` ничего не меняется.
- **Localhost открыт (frictionless).** Подключения с `127.0.0.1` / `::1` проходят без входа —
  даже если секрет задан. Это позволяет работать локально без лишних движений. Чтобы закрыть и
  localhost тоже, добавьте флаг `--auth`.
- **Electron никогда не гейтит.** Десктопная сборка (Electron) доступна только через саму
  оболочку Electron, у которой уже есть собственный токен безопасности, поэтому проверка там не
  включается вообще.
- **Ворота закрываются и для WebSocket/RPC.** Проверка стоит не только на HTTP, но и на
  upgrade-запросе WebSocket — без валидной сессии соединение RPC отклоняется. HTTP-only защита
  была бы дырой.

## Как включить

Нужны две вещи: **секрет** (пароль/токен) и, при желании, **флаг** для гейтинга localhost.

### 1. Задать секрет

Вариант A — переменная окружения (самый простой):

```bash
export AI_FOCUSED_EDITOR_AUTH_SECRET='ваш-длинный-общий-секрет'
bun run start:browser
```

Вариант B — сохранённый **солёный хеш** (пароль не хранится в открытом виде). Задайте пароль
один раз — он будет захеширован (scrypt) и записан в `~/.ai-focused-editor/auth.json`:

```bash
bun run start:browser -- --auth-set-password 'ваш-пароль'
# в логах появится строка: [browser-auth] stored salted password hash at ~/.ai-focused-editor/auth.json
# после этого перезапустите БЕЗ этого флага
bun run start:browser
```

Можно задать оба варианта одновременно — подойдёт любой из совпавших секретов.

Дополнительно:

- `AI_FOCUSED_EDITOR_AUTH_TTL` — время жизни сессии в секундах (по умолчанию 7 дней).
- `AI_FOCUSED_EDITOR_AUTH_HOME` — переопределяет каталог `~/.ai-focused-editor`.

### 2. (Опционально) закрыть и localhost

```bash
bun run start:browser -- --auth
```

С флагом `--auth` вход требуется для всех подключений, включая `127.0.0.1`. Без флага localhost
остаётся открытым. Если передать `--auth`, но не задать секрет/хеш, ворота останутся
**выключенными** (чтобы не заблокировать доступ полностью) — в логах будет предупреждение.

## Вход

Откройте редактор в браузере. Если ворота включены и сессии нет, вместо приложения покажется
страница входа. Введите пароль/токен — при успехе ставится подписанная, `HttpOnly`,
`SameSite=Lax` cookie-сессия, и происходит переход в редактор.

## Вход по QR (с телефона)

Из уже авторизованной сессии (или с localhost) выполните команду **«AI Focused Editor: Показать
QR для входа»** (палитра команд). Появится диалог с QR-кодом и ссылкой под ним. Отсканируйте QR
телефоном, подключённым к той же сети, — телефон получит собственную сессию.

QR-ссылка использует **одноразовый токен** со сроком жизни ~2 минуты. После первого перехода
токен становится недействительным.

## TLS через реверс-прокси (Caddy)

Сам редактор отдаёт cookie без флага `Secure` по обычному HTTP; при работе через прокси с TLS
редактор видит заголовок `X-Forwarded-Proto: https` и ставит cookie уже как `Secure`. Поднять
HTTPS проще всего через [Caddy](https://caddyproxy.com) — одна строка в `Caddyfile`:

```caddy
editor.example.com {
    reverse_proxy 127.0.0.1:3000
}
```

Caddy сам получит и продлит сертификат Let's Encrypt, а редактор запускайте на `127.0.0.1:3000`
с заданным секретом. Для сети рекомендуется всегда ставить TLS-прокси перед редактором, чтобы
секрет и cookie не ходили по сети в открытом виде.

## Как это устроено (кратко)

- **HTTP:** Express-middleware, зарегистрированное через `EarlyExpressMiddleware` — выполняется
  раньше статики и RPC. Без валидной cookie неавторизованному навигационному запросу отдаётся
  страница входа (200), а XHR — `401`.
- **WebSocket/RPC:** `WsRequestValidatorContribution.allowWsUpgrade` — отклоняет upgrade без
  валидной cookie.
- **Крипто:** пароль хешируется `scrypt`, сверяется через `timingSafeEqual`; cookie — это
  `HMAC-SHA256` идентификатора сессии на серверном секрете, который меняется при перезапуске
  (перезапуск инвалидирует старые сессии).
